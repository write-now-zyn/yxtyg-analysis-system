# AGENTS.md

适用范围：仓库根目录 `/Users/clown/Documents/code/yxtyg` 及其全部子目录。

本文件用于约束后续在本仓库内进行的代码阅读、修改、重构和新增功能。除非用户明确要求偏离，否则应默认遵守。

## 1. 项目结构

- `yxtyg-admin`：Spring Boot 后端
- `yxtyg-web`：Vue 2 + Element UI 前端
- `docs`：初始化文档、SQL 脚本、开发规范
- `data/vectors`：本地向量数据

## 2. 总体原则

- 优先做小步、可验证、低回归风险的改动，不做无边界重写
- 修问题时先复用现有模式，再做抽象；只有重复达到 2 处以上再考虑提炼公共模块
- 不允许为了“看起来更现代”而引入与当前技术栈不匹配的框架或重型依赖
- 不允许无理由混用中英文命名风格；应与所在模块既有风格保持一致
- 修改接口、字段、路由、SQL 结构时，必须同步检查前后端联动影响

## 3. 后端代码规范

适用目录：`yxtyg-admin`

### 3.1 基本约束

- 保持当前单体 Spring Boot 架构，不擅自引入微服务拆分、Spring Cloud、消息中间件等新基础设施
- 优先沿用现有分层：`controller`、`service`、`service/impl`、`mapper`、`entity`、`dto`、`vo`
- Controller 只负责参数接收、结果返回和基础校验，不写大段业务逻辑
- 业务逻辑放在 `service/impl`
- 数据访问逻辑放在 `mapper` 和对应 XML，不要把复杂 SQL 散落到业务层

### 3.2 接口与返回

- 接口返回统一复用现有 `Result` / `PageResult`
- 新接口路径应按业务域归类到现有前缀，例如 `/api/workorder`、`/api/review`
- 分页查询优先延续当前分页参数和返回结构，避免前端额外适配

### 3.3 数据对象

- `DTO` 用于接收入参
- `VO` 用于返回前端
- `Entity` 对应数据库表，不直接承载前端展示拼装字段，展示字段优先放 `VO`
- 不要把一个对象同时当作 `Entity + DTO + VO` 使用

### 3.4 异常与事务

- 业务异常优先使用现有 `BusinessException`
- 涉及多表写入、状态一致性的服务方法要显式评估是否需要 `@Transactional`
- 不要滥用 `catch` 吞异常；确需降级时要保留明确的失败原因

### 3.5 SQL 与配置

- 新增表结构或字段变更时，必须同步更新初始化 SQL，优先补到 `yxtyg-admin/src/main/resources/db/init.sql`；如果项目实际以其他初始化脚本为准，也必须同步更新对应初始化脚本，不能只补增量脚本
- 新增表结构或字段变更时，必须补充 SQL 脚本到 `docs` 或 `src/main/resources/sql` / `db`
- 不能只改 Java 代码不补初始化或升级脚本
- 配置优先放 `application.yml` 或数据库配置表，避免硬编码 URL、密钥、路径
- 复杂 SQL 尽量写在对应的 `mapper.xml` 中统一维护，不要把长 SQL、复杂条件拼接、大段多表查询直接散落在 Java 代码里

### 3.6 后端改动时的检查项

- 是否影响现有前端请求字段
- 是否已经同步更新 `init.sql` 和必要的升级脚本
- 是否影响数据库初始化或升级路径
- 是否影响导出、图片读取、AI 调用、向量功能这类已有链路

## 4. 前端代码规范

适用目录：`yxtyg-web`

### 4.1 基本约束

- 保持 Vue 2 + Options API + Element UI 的现有技术栈，不擅自升级到 Vue 3 或引入新的状态管理体系
- 页面逻辑优先保持单文件组件风格，但重复布局、重复请求流程、重复展示块应逐步提取公共模块
- 不允许在多个页面复制粘贴同一段布局 CSS 而不做抽象

### 4.2 页面布局规范

- 页面根容器统一使用 `.page-container`
- 页面外层留白统一由 `App.vue` 的 `.app-main` 控制
- 页面组件内不要再给 `.page-container` 追加额外 `padding`
- 一级内容块优先使用 `el-card.page-card`
- 卡片头部统一使用：
  - `.page-card__header`
  - `.page-card__title`
  - `.page-card__actions`
- 通用布局样式统一维护在 `yxtyg-web/src/styles/layout.css`

布局规范的详细说明见：

- `docs/frontend-layout-guidelines.md`

### 4.3 样式职责边界

- 全局布局规则放公共样式文件，不放到单页 `scoped style` 里重复定义
- 页面内样式只处理业务特有结构，例如图片区、搜索条件宽度、状态标签、表格单元格展示
- 尽量减少内联样式；涉及 `padding`、`margin`、`background`、`display` 的通用布局样式优先提取到 CSS 类

### 4.4 组件与复用

- 多个页面重复出现的结构，优先考虑抽取为公共组件或公共样式
- 优先抽取这些重复模式：
  - 搜索卡片
  - 表格卡片
  - 页面头部
  - 导入/导出按钮区
- 抽象前先确认至少有 2 个以上页面存在近似重复，避免过度封装

### 4.5 请求与数据处理

- 所有 HTTP 请求统一走 `src/utils/request.js`
- 接口定义统一放 `src/api/*.js`
- 页面组件内不要直接拼接重复的 axios 请求
- 前端展示字段转换逻辑尽量收敛到页面 methods 或小型工具函数，避免模板中嵌套复杂表达式

### 4.6 前端改动时的检查项

- 页面间距、卡片头部、按钮区样式是否与同类页面一致
- 菜单、路由、接口路径是否同步
- 是否破坏 `/api` 代理约定
- 是否引入了与现有 Element UI 风格明显冲突的组件写法

## 5. 文档与规范同步

- 改动如果改变了初始化步骤、端口、脚本、数据库结构、布局规范，应同步更新 `docs`
- 新增通用前端约束，优先补到：
  - `docs/frontend-layout-guidelines.md`
  - 本 `AGENTS.md`

## 6. 禁止事项

- 不要随意修改已存在接口返回结构，除非同时完成前端适配
- 不要把临时调试代码、测试账号、固定密钥提交进仓库
- 不要在页面里继续新增与全局规则冲突的 `.page-container { padding: ... }`
- 不要在后端把展示拼接、格式化文本、复杂 JSON 解析无边界堆进 Controller
- 不要无依据地“大面积统一风格”，导致用户已熟悉的页面行为回归

## 7. 推荐工作方式

- 先查现有实现，再决定复用还是抽象
- 先做必要修复，再做顺手的低风险收敛
- 产出应尽量满足：可运行、可验证、可维护、可追踪
