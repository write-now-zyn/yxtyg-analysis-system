# 工单模块开发文档

## 一、需求文档

### 1.1 需求背景

工单数据来源于一线体验官系统外部导入，需要将工单Excel表格导入系统进行统一管理和查询。

### 1.2 功能需求

#### 1.2.1 工单导入
- 支持导入工单Excel模板文件
- Excel模板包含13个列字段：BOMC流水号、申告号码、处理单位、处理人、申告地市、创建时间、发起人姓名、处理满意度评分、派单满意度评分、不满意原因、状态、解决方案历史、工单内容

#### 1.2.2 导入规则
1. 当前导入工单如果数据库中已存在相同流水号且相同状态的，就跳过，不更新数据库
2. 如果导入文件中流水号在数据库中存在，但是状态不一致，那就按照流水号更新全部其它字段

#### 1.2.3 解决方案历史解析
- 解决方案历史以竖线分隔，格式如：`处理人处理意见: 意见内容|处理人处理意见: 意见内容`
- 需要解析并存储到独立的解决方案历史表

#### 1.2.4 工单列表
- 列表只展示关键字段：问题编号（流水号）、发起人、地市、工单内容、状态
- 支持按问题编号、发起人、地市、状态、关键词进行搜索
- 支持分页查询

#### 1.2.5 工单详情
- 点击工单可查看全部字段
- 解决方案历史默认折叠，点击可展开查看
- 使用时间线组件展示解决方案历史

#### 1.2.6 删除功能
- 支持单条删除
- 支持批量删除
- 删除工单时同时删除关联的解决方案历史

## 二、设计文档

### 2.1 数据库设计

#### 2.1.1 工单主表（t_work_order）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | BIGINT | 主键ID，自增 |
| order_no | VARCHAR(50) | BOMC流水号 |
| declare_phone | VARCHAR(20) | 申告电话 |
| handle_unit | VARCHAR(100) | 处理单位 |
| handler_name | VARCHAR(50) | 处理人 |
| city | VARCHAR(50) | 申告地市 |
| create_time_str | VARCHAR(50) | 创建时间（原始字符串） |
| initiator_name | VARCHAR(50) | 发起人姓名 |
| handle_satisfaction_score | VARCHAR(20) | 处理满意度评分 |
| dispatch_satisfaction_score | VARCHAR(20) | 派单满意度评分 |
| unsatisfied_reason | VARCHAR(500) | 不满意原因 |
| status | VARCHAR(50) | 状态 |
| solution_history_str | TEXT | 解决方案历史原始字符串 |
| content | TEXT | 工单内容 |
| created_at | DATETIME | 记录创建时间 |
| updated_at | DATETIME | 记录更新时间 |

**索引设计：**
- PRIMARY KEY: id
- UNIQUE KEY: uk_order_status (order_no, status)
- INDEX: idx_order_no, idx_city, idx_status, idx_create_time

#### 2.1.2 解决方案历史表（t_solution_history）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | BIGINT | 主键ID，自增 |
| work_order_id | BIGINT | 工单ID |
| order_no | VARCHAR(50) | BOMC流水号（冗余存储） |
| handler_name | VARCHAR(50) | 处理人姓名 |
| opinion_content | TEXT | 意见内容 |
| sort_order | INT | 排序序号 |
| created_at | DATETIME | 创建时间 |

**索引设计：**
- PRIMARY KEY: id
- INDEX: idx_work_order_id, idx_order_no

### 2.2 接口设计

| 接口路径 | 方法 | 说明 |
|----------|------|------|
| /api/workorder/list | GET | 分页查询工单列表 |
| /api/workorder/detail/{id} | GET | 获取工单详情 |
| /api/workorder/import | POST | 导入工单Excel |
| /api/workorder/{id} | DELETE | 删除工单 |
| /api/workorder/batchDelete | POST | 批量删除工单 |

### 2.3 类设计

#### 2.3.1 后端类结构

```
com.jscm.yxtyg
├── entity/
│   ├── WorkOrder.java           # 工单实体
│   └── SolutionHistory.java     # 解决方案历史实体
├── dto/
│   ├── WorkOrderExcelDTO.java   # Excel导入DTO
│   └── WorkOrderQueryDTO.java   # 查询DTO
├── vo/

│   └── WorkOrderDetailVO.java   # 详情VO
├── mapper/
│   ├── WorkOrderMapper.java
│   └── SolutionHistoryMapper.java
├── service/
│   ├── WorkOrderService.java
│   ├── SolutionHistoryService.java
│   └── impl/
│       ├── WorkOrderServiceImpl.java

├── controller/
│   └── WorkOrderController.java
```

#### 2.3.2 Excel解析类设计

`WorkOrderExcelParser` 是独立的解析类，便于后续修改列映射。主要方法：
- `parse(MultipartFile file)` - 解析Excel文件
- `parseSolutionHistory(String solutionHistoryStr, String orderNo)` - 解析解决方案历史字符串
